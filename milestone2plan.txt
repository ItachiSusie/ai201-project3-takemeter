# Milestone 2 Plan (Spec Before Annotation)

## Project scope
I will build a text classifier using public Reddit content from the Kaggle dataset:  
`https://www.kaggle.com/datasets/armitaraz/chatgpt-reddit`.

---

## 1) Community
**Chosen community:** ChatGPT-focused Reddit discussions contained in the Kaggle `chatgpt-reddit` dataset (primarily posts/comments about ChatGPT usage, issues, and updates).

**Why this community:** It is a strong fit for classification because people discuss different intents in the same space: troubleshooting, prompt advice, product/news reactions, and sharing generated results. That makes the discourse varied in purpose and writing style, which is exactly what a useful classifier needs.

---

## 2) Labels (4 classes)
I will use 4 labels to keep boundaries clear while still capturing meaningful variation.

### `troubleshooting`
A post/comment where the main goal is to diagnose or fix a problem (errors, outages, account/access issues, broken behavior, or unexpected model output seen as a bug).

**Example A:**  
"Is anyone else getting 'Something went wrong' every time I send a message? I already cleared cache and switched browsers."

**Example B:**  
"My GPT keeps ignoring uploaded files since this morning. Did OpenAI change something or is this a known issue?"

### `prompting_help`
A post/comment asking for or giving practical guidance on how to write prompts, structure instructions, or improve response quality.

**Example A:**  
"What prompt format do you use to make ChatGPT return consistent JSON every time?"

**Example B:**  
"Try giving role + constraints + output schema in one prompt; it reduces hallucinations a lot."

### `news_policy_discussion`
A post/comment focused on updates, releases, pricing, policy, safety, company decisions, or broader implications rather than direct troubleshooting or prompting.

**Example A:**  
"The new pricing tier makes no sense for casual users. Why did they remove the old plan?"

**Example B:**  
"OpenAI's latest model release notes mention memory behavior changes; this could affect education use cases."

### `showcase_story`
A post/comment primarily sharing a result, workflow, experiment, or personal experience ("look what I built," "here's what happened when I tried X").

**Example A:**  
"I used ChatGPT + Sheets to automate weekly status reports for my team. Saved me ~3 hours this week."

**Example B:**  
"Built a small RPG quest generator with custom instructions; here are a few outputs that were surprisingly good."

---

## 3) Hard edge cases and annotation rule
### Ambiguous type A: `troubleshooting` vs `prompting_help`
Posts like: "Why does ChatGPT ignore my formatting instructions?" can be either a bug report or prompt-design issue.

**Rule:**  
If the author frames it as system malfunction/outage/account/platform fault, label `troubleshooting`.  
If the author asks how to phrase instructions better or improve prompt strategy, label `prompting_help`.

### Ambiguous type B: `news_policy_discussion` vs `showcase_story`
Posts like: "After the new model update, my workflow improved a lot" blend update commentary and personal use.

**Rule:**  
If the central point is discussing the update/policy itself, label `news_policy_discussion`.  
If the central point is the author's workflow/output story, label `showcase_story`.

### Ambiguous type C: `showcase_story` vs `prompting_help`
Posts like: "Here's my prompt that gets great results" can be sharing or instruction.

**Rule:**  
If the post is mainly "what I made / what happened," label `showcase_story`.  
If it is mainly "how you should do this," label `prompting_help`.

When uncertain, I will record a note in the CSV `notes` column explaining which competing labels I considered and why I chose one.

---

## 4) Data collection plan
**Primary source:** Kaggle `chatgpt-reddit` dataset only (public Reddit content).  
**Initial target:** collect and label **220** items, then keep at least **200** final clean items.

**Target distribution goal (before finalizing 200):**
- troubleshooting: 50-60
- prompting_help: 50-60
- news_policy_discussion: 50-60
- showcase_story: 50-60

This keeps any single label below 70% and gives enough examples per class for meaningful evaluation.

**If a label is underrepresented after first 200 reviewed items:**
1. Pull additional candidates from the same Kaggle dataset using keyword-guided sampling for that label.
2. Label extra items until each minority class has enough volume and no class exceeds 70%.
3. Keep only one final CSV (not pre-split), minimum 200 rows.

---

## 5) Evaluation metrics
I will evaluate with:

1. **Macro F1 (primary):** treats all classes equally and avoids majority-class bias.
2. **Per-class Precision/Recall/F1:** shows whether any specific label is weak.
3. **Confusion Matrix:** identifies which label pairs are repeatedly confused.
4. **Accuracy (secondary):** useful summary metric, but not sufficient alone.

**Why not accuracy alone:** A model could be "accurate" by over-predicting the largest class. Macro F1 + per-class metrics are required to validate real usefulness for multi-class community moderation/support use.

---

## 6) Definition of success
The classifier is useful only if it is reliable across all labels, not just overall.

**Good enough (minimum for practical use):**
- Macro F1 >= 0.70
- No class F1 below 0.60
- Accuracy >= 0.72

**Strong/deployment-ready target:**
- Macro F1 >= 0.78
- No class F1 below 0.70
- No single confusion pair dominates errors (checked in confusion matrix)

If these thresholds are not met, I will revise label rules and/or add better-balanced data before claiming readiness.

---

## 7) AI Tool Plan
### A) Label stress-testing (before full annotation)
I will paste my label definitions + edge-case rules into an LLM and ask for **5-10 boundary posts** that intentionally blur two labels.

If I cannot classify several of them cleanly, I will tighten definitions before annotating the 200 examples.

### B) Annotation assistance
I may use an LLM to pre-label a batch, then manually review all labels.

If used, I will track this with a column such as `prelabeled_by_llm` (yes/no) and keep corrections in `notes` for transparency in AI usage disclosure.

### C) Failure analysis
After model evaluation, I will provide wrong predictions to an LLM and ask for error-pattern hypotheses (e.g., recurring confusion pairs, wording patterns).

I will verify each suggested pattern manually against raw examples before writing conclusions.

---

## Evaluation-plan sanity check
These criteria are objective: each has a numeric threshold or explicit confusion-matrix check, so I can clearly determine whether the project met success criteria at the end.
